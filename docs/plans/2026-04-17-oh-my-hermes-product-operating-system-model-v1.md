# oh-my-hermes — Product Operating System Model v1

## 1. Purpose
Define the **stable recurring operating cycle** of oh-my-hermes as a living product.

This layer answers:
- how the product runs as an ongoing system
- how learning, governance, execution, and publication fit together
- what the steady-state operating loop of the product actually is

This is not the execution loop for a single user project.
It is the operating loop for the harness product itself.

---

## 2. Core design idea
Now that the product has:
- a runtime contract
- a self-evolution governance model
- an evidence model
- a promotion policy
- a milestone governance model
- a constitution

it needs one more thing:

**a stable operating system for how all these layers run together over time.**

Without this, the design stack remains valid but static.
With it, the product becomes a continuously operating governed system.

---

## 3. Product OS definition
The Product Operating System is the recurring meta-loop by which oh-my-hermes:
1. takes in work
2. executes through the harness
3. learns from execution
4. governs what was learned
5. formalizes the right changes
6. republishes its own product state

This is the layer that turns the product from a collection of design doctrines into an ongoing machine.

---

## 4. The six recurring operating phases

### Phase 1 — Intake of work
Purpose:
Convert external demand into bounded product-relevant work.

Examples:
- a user asks for a new capability
- a sample project reveals a gap
- repeated backlog pressure indicates design debt
- a milestone frontier becomes clear

Question answered:
**What work should the product now absorb?**

---

### Phase 2 — Controlled execution
Purpose:
Run the harness against real work or real validation contexts.

Meaning:
- sample projects
- internal product refinement work
- dogfood execution
- governance hardening passes

Question answered:
**What happened when the product was exercised against reality?**

---

### Phase 3 — Signal capture
Purpose:
Capture execution outcomes as structured signals, not raw noise.

Examples:
- failures
- repeated recovery patterns
- repeated review findings
- repeated contract mismatches
- operator-facing ambiguity
- evidence of missing policy

Question answered:
**What meaningful signals came out of execution?**

---

### Phase 4 — Governance and interpretation
Purpose:
Interpret signals through evidence and governance layers.

Outputs:
- accepted now
- backlog
- deferred
- user decision required

Question answered:
**What do these signals mean for the product?**

---

### Phase 5 — Formalization
Purpose:
Turn governed learning into official product state.

Examples:
- promote policy
- update constitutional interpretation
- add new design layer
- keep some themes in backlog
- publish milestone boundary

Question answered:
**What is now officially true about the product?**

---

### Phase 6 — Publication and restart
Purpose:
Publish the newly formalized state into canonical history and restart from the new baseline.

Meaning:
- commit
- push
- make the new state the baseline for future learning

Question answered:
**What is the next operating baseline?**

---

## 5. Canonical operating loop
The product OS loop is:

1. absorb work
2. exercise the product
3. capture signal
4. interpret signal
5. formalize product truth
6. publish baseline
7. repeat

This is the long-running product loop.

It is distinct from the inner harness loop of:
- intake
- plan
- execute
- verify
- accept

So there are now two loops:

### Inner loop
Delivers a project or bounded task.

### Outer loop
Evolves the harness product itself.

That distinction is central.

---

## 6. Why the two-loop model matters
Without separating these loops, the product confuses:
- doing work for the user
with
- learning how the product should evolve

That confusion creates two failures:

### Failure A — no product learning
The harness keeps delivering but never upgrades its own operating model.

### Failure B — unstable product mutation
The harness tries to redefine itself in the middle of serving current work.

The two-loop model prevents both:
- inner loop = delivery
- outer loop = governed product evolution

---

## 7. Product OS surfaces
The operating system should be understood as managing five persistent surfaces.

### 1. Demand surface
What new work is entering the system?

### 2. Execution surface
What happened when the system ran?

### 3. Learning surface
What did the system infer from that execution?

### 4. Governance surface
What is allowed to change now?

### 5. Publication surface
What has entered canonical product history?

These surfaces define the ongoing life of the product.

---

## 8. Operating rhythm
The product OS should run in two rhythms.

### Continuous rhythm
Always on:
- execution
- signal capture
- backlog accumulation
- low-level interpretation

### Boundary rhythm
Triggered at milestone boundaries:
- promotion decisions
- formalization
- constitutional checks
- canonical publication

This creates a healthy asymmetry:
- learning is continuous
- lawmaking is staged

That is the right operating rhythm for a governed self-improving product.

---

## 9. Constitutional role inside the Product OS
The constitution should be understood as the outermost guardrail around the Product OS.

Meaning:
- the OS can evolve policies
- the OS can evolve heuristics
- the OS can evolve governance detail
- but the OS may not silently cross constitutional boundaries

So the Product OS is not unconstrained recursion.
It is a bounded operating cycle inside constitutional limits.

---

## 10. Product OS outputs
A healthy Product OS should continuously produce four outputs.

### Output A — delivered work
Real external value through the inner loop.

### Output B — backlog intelligence
Structured unresolved learning.

### Output C — promoted product law
New accepted rules and design layers.

### Output D — canonical milestone history
Legible GitHub-visible product evolution.

If one of these outputs disappears, the system is unhealthy.

---

## 11. Product OS failure modes
Recommended failure modes to watch.

### Failure mode 1 — execution-only mode
The system keeps delivering but never matures its own operating model.

### Failure mode 2 — reflection-only mode
The system keeps theorizing about itself but stops creating real execution pressure.

### Failure mode 3 — drift mode
The system keeps changing rules without milestone discipline or constitutional guardrails.

### Failure mode 4 — backlog swamp
The system learns continuously but never promotes anything.

### Failure mode 5 — brittle law mode
The system formalizes too aggressively and loses flexibility.

The Product OS exists to keep the product between these extremes.

---

## 12. Steady-state design goal
The goal is not to maximize execution volume.
The goal is not to maximize self-editing speed.

The true steady-state goal is:

**reliable delivery plus governed product improvement plus legible history**

That is the right operating target for oh-my-hermes.

---

## 13. Current Product OS recommendation
At the current stage of oh-my-hermes, the recommended operating sequence is:

1. use sample projects to generate real execution pressure
2. capture resulting governance and evidence lessons
3. formalize only milestone-grade product layers
4. keep canonical history clean and product-legible
5. gradually shift from design stack completion toward reusable operating substrate

Meaning:
The product is still in architecture-consolidation mode, not yet in broad execution-scale mode.
